# v0.5.1 — Fix self-tail feedback loop; pin image tags; release tooling fix

**Status:** shipped, tagged `v0.5.1`.

## Goal

`make k8s-apply`ing [v0.5.0](v0.5.0.md) surfaced a live bug: grepod's own
pod stuck in a tight loop logging `batch queue full, dropping line for
pod=grepod-... container=grepod`, over and over. Alongside the fix, two
adjacent gaps got cleaned up in the same pass: both manifest flavors
defaulted to a floating `:latest` image tag instead of a pinned version,
and `make release` had regressed back to an auto-`--amend`-and-force-push
flow that [v0.4.1](v0.4.1.md)/[v0.4.2](v0.4.2.md) had already deliberately
moved away from.

## Delivered scope

- **Self-tail feedback loop, root fix**: grepod had no special case for
  its own pod, so it tailed itself like any other container in its
  namespace. `storage.BatchQueue.Enqueue` logs a warning when its internal
  channel is full — written to grepod's own stdout, which grepod would
  then tail back in and try to enqueue. If the queue was still full, that
  attempt logged another warning, which got tailed back in, indefinitely —
  self-sustaining regardless of what originally filled the queue, and
  never recovering on its own. Fixed by threading `POD_NAME` (already read
  in `cmd/server/main.go` via the Downward API) into a new `selfPod`
  parameter on `tailer.NewManager`; `reconcilePod` now returns immediately
  for `pod.Name == selfPod`, before starting any container's tailer
  goroutine. See [DESIGN/02](../DESIGN/02_design_tailer.md#never-tailing-itself-v051).
- **Rate-limited "queue full" warning, defense in depth**: even with
  self-tail excluded, a genuine sustained overload from some *other* pod
  could still flood grepod's own log output with one warning per dropped
  line. `BatchQueue.recordDrop` now logs at most one summarizing warning
  (with a `count`) per 5s window instead of one per drop, using two
  atomics rather than a mutex so `Enqueue`'s existing never-block guarantee
  holds on this path too. See
  [DESIGN/03](../DESIGN/03_design_storage.md#never-flooding-on-a-full-queue-v051).
- **Image tags pinned to a version, not `:latest`**: `k8s/20-deployment.yaml`
  now hardcodes `ghcr.io/teochenglim/grepod:<version>` (no `v` prefix —
  `docker/metadata-action`'s `type=semver,pattern={{version}}` strips the
  git tag's leading `v` when publishing to GHCR); `helm/values.yaml`'s
  `image.tag` defaults to `""`, falling back to `.Chart.AppVersion` in
  `helm/templates/deployment.yaml`, with `helm/Chart.yaml`'s `appVersion`
  now kept current. Deliberate: `:latest` meant a redeploy could silently
  pick up a newer image than whatever was actually tested.
- **`make bump` now also regex-updates `helm/Chart.yaml`'s `appVersion`
  and `k8s/20-deployment.yaml`'s image tag** alongside `VERSION` (`sed
  -i.bak ... && rm -f *.bak`, portable across BSD/GNU `sed`), so a single
  `make release VERSION=x.y.z` keeps all three in sync instead of three
  manual edits.
- **`make release`'s auto-`--amend`-and-`--force-with-lease`-push removed**:
  somewhere after [v0.4.2](v0.4.2.md) shipped, `release` had regressed to
  `git add VERSION && git commit --amend --no-edit && git push
  --force-with-lease origin HEAD` — exactly the auto-commit behavior
  v0.4.1 removed for good reason (a generic bump-commit message ends up on
  the tag, visible in GitHub Actions' run list) and a force-push nobody
  asked it to do. Restored to v0.4.2's actual shipped design: `bump` (now
  covering all three files), plain `git push origin HEAD`, `git tag`,
  `git push origin <tag>` — no commit, no `--amend`, no force-push. `make
  release`'s help banner and `RELEASE.md`'s "Cutting a release" section
  (which had drifted to describe the *pre*-v0.4.2 flow, missing `git push
  origin HEAD` entirely) both updated to match what actually ships now.

## Bugs found and fixed

- The self-tail feedback loop above — see "Delivered scope."
- `make release`'s auto-commit-and-force-push regression — see "Delivered
  scope." Root cause not identified (no commit in this repo's history
  visibly reintroduces it), so treat this release's Makefile as the
  source of truth going forward, not whatever the diff against the
  previous commit implies.

## Verified

`go build`/`go vet`/`gofmt`/`go test -race -cover` (all packages green;
new coverage: `TestReconcilePod_NeverTailsSelfPod`,
`TestReconcilePod_EmptySelfPodDoesNotExcludeAnything`,
`TestBatchQueue_RateLimitsFullWarning`,
`TestBatchQueue_WarnsAgainAfterRateLimitWindow`). `make bump
VERSION=0.5.1` run for real and confirmed idempotent/correct via `git
diff` on all three files. `helm lint` (clean) and `make helm-template`
confirmed the rendered `Deployment` resolves
`ghcr.io/teochenglim/grepod:0.5.1`; `kubectl kustomize k8s/` confirmed the
same tag renders from the plain manifests. No cluster available in this
environment to re-run `make k8s-apply` and confirm the warning loop is
actually gone live — the fix is covered by the two new `tailer` tests
(`reconcilePod` never starting a tailer for `selfPod`) plus the two new
`storage` tests (rate limiting), but a human should redeploy and confirm
no `batch queue full` loop before calling this fully closed.
