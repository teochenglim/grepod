# v1.0.0 — Read/write timeouts, SQLite write consolidation, then stabilize & commit to compatibility

**Status:** shipped, tagged `v1.0.0`.

Originally two releases — a `v0.8.0` (timeouts/write-consolidation) ahead
of a code-free `v1.0.0` (compatibility commitment). Folded into one here:
the two turned out to look like the same kind of release at a glance,
and there's no real reason to make and tag an intermediate 0.8.0 when
1.0.0 was already the very next stop on the roadmap.

## Goal (read/write timeouts)

`cmd/server/main.go`'s `http.Server` only sets `ReadHeaderTimeout: 10 *
time.Second`. Go's own `http.Server` docs call this out explicitly: without
`ReadTimeout`/`WriteTimeout`/`IdleTimeout`, a slow client can still hang a
connection indefinitely after headers are read (a slow request body, or a
slow reader on the response side) — `ReadHeaderTimeout` alone doesn't cover
that. Separately, `storage.Store.Search` takes no `context.Context` at all
today: it calls `sql.DB.Query`/`sql.DB.Exec`, not the `...Context` variants,
so a request's timeout/cancellation never actually reaches the database —
an expensive cross-shard query has nothing bounding it but whatever the new
`WriteTimeout` eventually enforces from the outside.

## Delivered scope (read/write timeouts)

- `cmd/server/main.go`'s `http.Server` now sets `ReadTimeout`
  (`HTTP_READ_TIMEOUT`, default 15s), `WriteTimeout`
  (`HTTP_WRITE_TIMEOUT`, default 30s — headroom for a genuinely large
  cross-shard search), and `IdleTimeout` (`HTTP_IDLE_TIMEOUT`, default
  120s), alongside the pre-existing `ReadHeaderTimeout: 10s`.
- `storage.Store.Search(ctx, opts)` and `storage.Store.InsertBatch(ctx,
  lines)` both check `ctx.Err()` up front and use `QueryContext`/
  `ExecContext`/`BeginTx`/`PrepareContext` throughout — a request's
  deadline or an early client disconnect now actually cancels the
  underlying SQLite operation instead of it running to completion
  regardless. `internal/api.Handler.handleSearch` passes `r.Context()`.
- **`/api/tail` exempted from `WriteTimeout`, as flagged ahead of time
  in [DESIGN/04](../DESIGN/04_design_api/02_tail_and_known.md#apitail-v040)**:
  `handleTail` calls
  `http.NewResponseController(w).SetWriteDeadline(time.Time{})` right
  after establishing the connection, clearing the deadline for that one
  long-lived connection without weakening `WriteTimeout` for every other
  (genuinely bounded) route.
- **Decided: yes, `BatchQueue`'s writer path gets a bounded per-flush
  context too.** `flush()` now wraps each `InsertBatch` call in
  `context.WithTimeout(context.Background(), insertTimeout)`
  (`INSERT_TIMEOUT`, default 30s) — a self-imposed backstop against one
  pathological shard write (stuck disk, WAL corruption) stalling the
  whole ingestion pipeline behind it forever. Deliberately generous:
  DESIGN/05's benchmarks put a normal 200-line flush at ~1.5ms, four
  orders of magnitude under this — it's a hang detector, not a latency
  target.
- New tests: `TestSearch_CancelledContextReturnsPromptly`,
  `TestInsertBatch_CancelledContextReturnsError` (`internal/storage`) —
  both assert `errors.Is(err, context.Canceled)` against an
  already-cancelled context, per the original plan.
- [DESIGN/03](../DESIGN/03_design_storage.md#context-bounded-queries-v080)
  and [DESIGN/04](../DESIGN/04_design_api/01_search.md#timeouts-and-client-disconnects-v100)
  document the new timeout values and the context-cancellation behavior;
  [DESIGN/04's tail section](../DESIGN/04_design_api/02_tail_and_known.md#apitail-v040)
  updated from "flagged, not yet fixed" to "here's how it's exempted."

## Goal (write-side SQLite concurrency)

SQLite tolerates one writer at a time well, but not a high rate of small,
frequent write transactions from concurrent goroutines — `getOrOpenDB`
already pins `SetMaxOpenConns(1)` per shard for exactly this reason.
`BatchQueue`'s current defaults (`BATCH_SIZE=200` or `BATCH_INTERVAL=500ms`,
whichever first) still flush fairly often under a busy namespace. Widen
the consolidation window so more lines land in a single transaction.

## Delivered scope (write-side SQLite concurrency)

- Default flush cadence moved from 500ms to 15s
  (`defaultBatchInterval`, `BATCH_INTERVAL` env var) — fewer, larger
  transactions per shard. `BATCH_SIZE` (default 200) remains the other
  trigger, so a busy namespace still flushes as soon as the buffer fills,
  regardless of the interval.
- **Re-checked `BatchQueue.Enqueue`'s channel capacity (`size*4`) against
  the new interval — concluded no change needed, and wrote down why**
  (see
  [DESIGN/03](../DESIGN/03_design_storage.md#context-bounded-queries-v080)):
  the channel only has to absorb lines arriving while `run()` is blocked
  inside one synchronous `flush()` call, which is bounded by `BATCH_SIZE`
  and that flush's own duration (~1.5ms at the default size) — neither of
  which `BATCH_INTERVAL` touches. `BATCH_INTERVAL` only matters for
  namespaces quiet enough to never hit `BATCH_SIZE` within the interval,
  and those never approach `size*4` lines in flight either way.
- Latency trade-off documented in
  [DESIGN/03](../DESIGN/03_design_storage.md#context-bounded-queries-v080):
  a line enqueued right after a flush on a quiet container is now
  searchable up to ~15s later instead of ~500ms later — expected, not a
  bug.
- New tests: `TestNewBatchQueue_DefaultsIntervalTo15s`,
  `TestNewBatchQueue_DefaultsInsertTimeoutTo30s` (`internal/storage`).
  `TestBatchQueue_FlushesOnSize` (pre-existing) already used an explicit
  1-hour interval, so it continued exercising the size-triggered path
  independently of the new default without needing changes.

## Verified (read/write timeouts + write consolidation)

`gofmt -l .` (clean), `go vet ./...`, `go build ./...`, and
`go test ./... -race -count=1` (all packages green) — includes every
pre-existing test updated for `Search`/`InsertBatch`'s new `ctx` parameter
across `internal/storage` and `internal/api`, plus the 4 new tests listed
above. `helm lint helm/` and `helm template` (rendering the new
`INSERT_TIMEOUT`/`HTTP_*_TIMEOUT` ConfigMap keys) both clean. Not verified
against a real cluster or a real slow/disconnecting client in this
session — the `/api/tail` write-deadline exemption in particular is
`net/http`-documented standard behavior
(`http.ResponseController.SetWriteDeadline`), not something this session
could observe end-to-end without a live long-running connection; worth an
explicit manual check (open a tail, wait past `HTTP_WRITE_TIMEOUT`,
confirm it's still alive) before or shortly after this ships.

## Goal (stabilize & commit to compatibility)

Not a features release — a promise release, once the above lands.
Everything in scope for 1.0 (tests, observability, live tail, restart-safe
tailing, RED metrics, and the timeout/consolidation work above) is done;
this is where grepod commits to not breaking the people using it.

## Delivered scope (stabilize & commit to compatibility)

  [COMPATIBILITY.md](../COMPATIBILITY.md) at repo root, linked from
  `README.md` and `CLAUDE.md`'s "Read these, in order" list. SemVer from
  here on. Covers every env var in `README.md`'s configuration table,
  every documented HTTP route's query params/response shape
  (`/api/search`, `/api/tail`, `/api/known`, `/healthz`/`/readyz`,
  `/metrics`'s metric names/types — widened from the original plan's
  "`/api/search` and `/api/tail`" to the full route table, since a
  compatibility promise with silent gaps isn't really a promise), and

  1:1, so the same stability applies there too, noted explicitly).

  namespace). Both considered for 1.0 and deliberately deferred, not
  overlooked.
