# v0.4.0 — Live tail backend

**Status:** shipped, tagged `v0.4.0`.

## Goal

Search answers "what happened." Live tail answers "what's happening right
now" — the `kubectl logs -f`-across-every-pod use case that started this
project, without waiting for a line to land in SQLite first. This release
is backend-only: the streaming endpoint and its fan-out, no UI yet (that's
[v0.5.0](v0.5.0.md)).

## Delivered scope

- **`GET /api/tail`** — Server-Sent Events (not WebSocket: no dependency
  needed, and grepod's tail is inherently one-directional). Optional
  `pod`/`container` (exact match) and `q` (case-insensitive substring)
  filters, applied per-connection. See
  [DESIGN/04](../DESIGN/04_design_api/02_tail_and_known.md#apitail-v040).
- **`storage.Broadcaster`** (`internal/storage/broadcast.go`) — fans each
  ingested line out to every live `/api/tail` subscriber, independent of
  `BatchQueue`'s eventual SQLite flush. Never blocks: a full subscriber
  has its *oldest* buffered line evicted to make room for the newest
  (opposite of `BatchQueue.Enqueue`'s drop-newest, deliberately — see
  [DESIGN/03](../DESIGN/03_design_storage.md#broadcaster-live-tail-fan-out)).
- **`fanoutSink`** (`cmd/server/main.go`) — the actual `tailer.Sink` now
  wired in, composing `BatchQueue.Enqueue` + `Broadcaster.Publish` per
  line. Neither `storage` nor `tailer` changed to know about the other's
  new collaborator; composition happens only in `main.go`.
- Extended the v0.2.0 test suite: `internal/storage/broadcast_test.go`
  (fan-out to N subscribers, drop-oldest-when-full, `Publish` never
  blocks, unsubscribe closes cleanly) and
  `internal/api/tail_test.go` (real `httptest.Server` + streaming HTTP
  client — SSE responses can't be exercised through
  `httptest.NewRecorder()` — covering event delivery, all three filters,
  and unsubscribe-on-client-disconnect).

## Bugs found and fixed (unrelated to live tail, found while reasoning about shutdown)

Prompted by a question about whether shutdown actually waits for tailer
goroutines to drain before flushing to SQLite — it didn't, on two counts:

- **SIGTERM was never caught.** `waitForShutdown` only called
  `signal.Notify` with `os.Interrupt` (SIGINT). SIGTERM is what Kubernetes
  actually sends on pod termination (`kubectl delete pod`, rolling
  updates, scale-down) — grepod never shut down gracefully in a cluster,
  only after `terminationGracePeriodSeconds` expired and it got SIGKILLed.
  Fixed: `signal.Notify(sigCh, os.Interrupt, syscall.SIGTERM)`.
- **Nothing waited for tailer goroutines to actually exit before closing
  the queue.** `cancel()` signals every `tailContainer` goroutine to stop,
  but `tailer.Manager.Run` returned immediately after `<-ctx.Done()`
  without waiting for them — so `main()`'s deferred `queue.Close()` could
  run while a straggling goroutine was still calling `Enqueue`,
  potentially dropping a handful of in-flight lines. Fixed: `Manager`
  gained a `sync.WaitGroup` tracking every spawned `tailContainer`
  goroutine; `Run` now blocks on `wg.Wait()` after `<-ctx.Done()` before
  returning, and `main.go` waits on a `mgrDone` channel (closed when
  `Run` returns) before proceeding to its deferred `queue.Close()`. This
  matters more now than before: v0.4.0 adds long-lived SSE connections,
  so shutdown correctness was worth nailing down before adding more
  concurrency on top, not after.
- Regression test: `TestManager_WaitGroupDrainsWhenContainerContextCancelled`
  exercises the `WaitGroup` mechanics directly (driving it through the
  full informer-backed `Run()` would be significantly more setup for the
  same guarantee).

## Verified

`go build`/`go vet`/`gofmt`/`go test -race -cover` (40/40), `docker
build`. `/api/tail` exercised end-to-end via a real `httptest.Server` and
streaming `http.Client` (not just `httptest.NewRecorder`, which can't
drive a live SSE response) — event delivery, per-field filtering, and
disconnect-unsubscribes all confirmed.
