# v0.4.0 — Live tail backend

**Status:** not started.

## Goal

Search answers "what happened." Live tail answers "what's happening right
now" — the `kubectl logs -f`-across-every-pod use case that started this
project, without waiting for a line to land in SQLite first. This release
is backend-only: the streaming endpoint and its fan-out, no UI yet (that's
[v0.5.0](v0.5.0.md)).

## Planned scope

- **`GET /api/tail`** — upgrades to a WebSocket (or falls back to SSE if
  that's simpler to keep dependency-free; decide at implementation time)
  and streams newly-ingested lines as they arrive, optionally filtered by
  `pod`, `container`, and/or a substring match — matching
  [DESIGN/04](../DESIGN/04_design_api.md)'s existing query-param style.
- **Fan-out from ingestion.** `internal/tailer.Manager.ingest` currently
  calls `Sink.Enqueue` once per line, straight into `storage.BatchQueue`.
  Live tail needs those same lines *before* they're batched/flushed to
  SQLite (that's seconds of latency `/api/tail` shouldn't have to eat).
  Introduce a broadcast point — e.g. a `pubsub`-style fan-out that
  `BatchQueue.Enqueue` (or a wrapping `Sink`) publishes to alongside its
  existing enqueue, so tailer/storage internals don't need to know about
  HTTP clients. Keep it non-blocking and best-effort per the same
  philosophy as `BatchQueue.Enqueue` — a slow or disconnected tail client
  must never back-pressure ingestion (see
  [DESIGN/03](../DESIGN/03_design_storage.md)).
- Bound per-connection memory (a slow client shouldn't accumulate an
  unbounded backlog — drop-oldest or disconnect past a buffer threshold).
- Extend the v0.2.0 test suite: fan-out delivers to N subscribers, a full
  subscriber buffer drops rather than blocks ingestion, filters narrow
  correctly.
