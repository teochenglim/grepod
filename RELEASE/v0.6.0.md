# v0.6.0 ‚Äî Hardening, perf pass, docs catch-up

**Status:** shipped, tagged `v0.6.0`.

## Goal

Everything functional for 1.0 exists by the end of v0.5.2. This release
is about making sure it holds up and that the docs actually describe what
shipped ‚Äî the last thing before committing to compatibility in
[v1.0.0](v1.0.0.md).

## Bugs found and fixed

### This release: the SQLite `ATTACH` limit

Running `BenchmarkSearch_AcrossShards` at 30 shards (not part of the
planned scope ‚Äî just pushing the benchmark past `RETENTION_DAYS`'
default of 7 to see how it scaled) surfaced a real, previously-unknown
bug: SQLite's default `SQLITE_MAX_ATTACHED` compile-time limit is **10**
databases per connection, confirmed empirically against
`modernc.org/sqlite` (an 11th `ATTACH` fails outright:
`"too many attached databases - max 10"`), with no portable way to raise
it from a pure-Go build. `Search`/`KnownPods` iterated shard dates
oldest-to-newest when building the `ATTACH` list; past the 10th, every
further `ATTACH` failed, was logged as a per-shard warning, and was
skipped ‚Äî meaning a date range needing more than 10 shards **silently
kept the oldest data and dropped the most recent**, backwards from what
any caller widening a search actually wants, with no signal in the API
response that anything was cut short. Reachable through completely
normal use (the UI's date pickers are free-form; `RETENTION_DAYS` is a
documented knob a busier deployment might set above 10), not just an
edge case.

Fixed: `existingShardDates` (new, shared by `Search` and `KnownPods`)
caps the shard list to the most recent `maxAttachedShards` (10) dates
when the range needs more, and logs one clear warning per call ‚Äî not one
per skipped shard ‚Äî naming how many were dropped and where the searched
range actually starts. Still a real limitation (a 30-day-wide search
only ever sees its most recent 10 days), but now documented and
predictable instead of silent and backwards. See
[DESIGN/03](../DESIGN/03_design_storage.md#store-daily-sharded-sqlite--fts5)
and [DESIGN/05](../DESIGN/05_design_performance.md#bug-found-the-attach-limit).
Fully removing the ceiling (batching `ATTACH` groups and merging in Go)
would need cursor pagination to work across batches too ‚Äî flagged for
whoever picks it up next, alongside [v1.0.0](v1.0.0.md)'s already-planned
SQLite write consolidation, since both touch the same connection-management
code.

### `v0.2.0` ‚Äî Architecture-driven test suite

Writing tests against documented behavior (not against whatever the code
happened to do) surfaced three real bugs, none previously caught because
nothing exercised these paths:

- **Cross-shard search was completely broken.** A schema-qualified table
  name (`d20260712.fts`) was passed into FTS5 auxiliary function
  arguments and `MATCH` ‚Äî FTS5 only resolves that inside the `FROM`
  clause, so every search failed outright with `no such column`/`no such
  table`. Fixed by aliasing the attached table back to unqualified `fts`
  in `FROM`.
- **Any search term containing FTS5 query-syntax characters (a leading
  hyphen, e.g. `flush-on-size-marker`) errored instead of searching** ‚Äî
  pod names, UUIDs, and stack traces routinely contain hyphens/colons.
  Fixed with `sanitizeMatchQuery`: each term quoted as its own FTS5
  phrase.
- **`BatchQueue.Close()` didn't actually wait for the final flush** and
  could drop a line enqueued immediately before `Close()` ‚Äî Go's
  pseudo-random `select` could pick `<-q.done` over a simultaneously
  ready `<-q.in`, and `Close()` returned before `run()`'s goroutine had
  actually finished flushing. Fixed: drain `q.in` before the final flush;
  `Close()` blocks on a `stopped` channel closed only after that flush
  completes.

### `v0.2.1` ‚Äî Security workflow fixes

`v0.2.0`'s tag push failed all three `security.yml` jobs. Reproduced
locally and fixed the actual findings rather than re-running CI and
hoping:

- **Semgrep**: `k8s/20-deployment.yaml`'s container had no
  `securityContext`, so `allowPrivilegeEscalation` defaulted to unset
  (effectively `true`). Added `allowPrivilegeEscalation: false` +
  `capabilities: {drop: ["ALL"]}` to both the plain manifest and the Helm
  template (Semgrep's YAML scanner skips Helm's `.tpl` syntax, so it
  wasn't flagged there, but the same gap existed).
- **Trivy filesystem**: 6 HIGH CVEs in transitive deps
  (`golang.org/x/net`, `golang.org/x/oauth2` via `k8s.io/client-go`).
  Bumped both; pulled `go.mod`'s `go` directive to `1.25.0` as a side
  effect.
- **Trivy image**: 10 HIGH Go-stdlib CVEs from an unpatched Go toolchain
  embedded in the compiled binary (detected from build info even in a
  distroless final image). Bumped the Dockerfile's build stage and CI's
  `GO_VERSION` to match.

### `v0.3.0` ‚Äî no bugs found (clean pass)

### `v0.4.0` ‚Äî Live tail backend

Found while reasoning about shutdown correctness, not live tail itself:

- **SIGTERM was never caught** ‚Äî only `os.Interrupt` was, so grepod never
  shut down gracefully in a cluster (`kubectl delete pod`, rolling
  updates), only after the grace period expired and it got SIGKILLed.
  Fixed: `signal.Notify` on both.
- **Nothing waited for tailer goroutines to actually exit before closing
  the queue** ‚Äî `cancel()` signals them to stop, but `Manager.Run`
  returned immediately without waiting, so a deferred `queue.Close()`
  could run while a straggling goroutine was still calling `Enqueue`.
  Fixed: `Manager` gained a `sync.WaitGroup`; `Run` blocks on it after
  `<-ctx.Done()`.

### `v0.4.1` ‚Äî Release tooling fix

Not a bug in the shipped product ‚Äî four design iterations on `make
release`'s own mechanics before landing on "bump inline, never commit,
never push `HEAD`," each a genuine correction: auto-commit meant a
generic message always ended up on the tag; requiring a pre-existing
`VERSION` match added friction without removing the `HEAD`-push issue;
dropping the push entirely still needed a separate `make bump` step.

### `v0.4.2` ‚Äî Release tooling fix

The exact failure mode v0.4.1's own note predicted happened for real on
v0.4.1's own release: its tagged commit was never pushed to `origin/main`
before `make release` ran, so GitHub's UI couldn't resolve the commit
message for the Actions run. Fixed: `git push origin HEAD` now runs
*before* creating the tag, not after (or never).

### `v0.5.0` ‚Äî no bugs found (clean pass)

### `v0.5.1` ‚Äî Fix self-tail feedback loop; release tooling regression

- **Self-tail feedback loop**: grepod tailed its own pod like any other
  container. `BatchQueue.Enqueue`'s "queue full" warning, written to
  grepod's own stdout, got tailed back in and retried ‚Äî if the queue was
  still full, that retry logged another warning, tailed back in again,
  indefinitely, regardless of what originally filled the queue. Fixed:
  `tailer.Manager` never tails `selfPod` (from `POD_NAME`).
  Paired with a rate-limited "queue full" warning (at most one summarized
  line per 5s) as defense in depth for a genuine sustained burst from
  another pod.
- **`make release` had regressed to an auto-`--amend`-and-force-push
  flow** ‚Äî exactly what v0.4.1 removed for good reason, root cause not
  identified in this repo's visible history. Restored to v0.4.2's actual
  shipped design.

### `v0.5.2` ‚Äî Fix permanent ingestion failure; browse without a keyword; fix unescaped log content

- **Permanent ingestion failure on a legacy-schema shard**: `CREATE
  VIRTUAL TABLE IF NOT EXISTS` no-ops against a shard predating v0.3.0's
  `level` column, and FTS5 rejects `ALTER TABLE` outright ‚Äî so that
  shard's original 5-column schema persisted forever and *every*
  subsequent insert into it failed, not just that shard's historical
  searchability. Fixed: `getOrOpenDB` detects and rebuilds (`DROP TABLE`
  + recreate) a legacy-schema shard on first write.
- **`make help` silently omitted targets with digits in their name**
  (`k8s-apply`, `k8s-delete`, `k8s-logs`) ‚Äî its target-listing regex had
  no `0-9` in its character class. Fixed.
- **Search required a keyword just to see anything**, discovered while
  confirming live tail worked but search looked "empty" (ingestion was
  actually current the whole time). Fixed: `q` is now optional ‚Äî browse
  mode.
- **Unescaped log content reaching the browser as real HTML** ‚Äî found
  while implementing browse mode, predated it. `snippet` was never
  HTML-escaped before `<mark>` highlighting was reintroduced via
  `innerHTML`; a log line containing real markup could execute as HTML ‚Äî
  a stored-XSS vector reachable by any pod's logs. Fixed: sentinel-byte
  markers + server-side `html.EscapeString`, so the only real markup that
  can ever reach the browser is highlighting grepod itself added.

## e2e smoke test: deferred

Evaluated wiring `scripts/local-test.sh`'s flow (kind cluster + build +
load + Helm install + hit `/healthz`/`/api/search`/`/api/tail`) into
`ci.yml` as a non-blocking job, per the planned scope. Decided to defer
rather than implement in this pass:

- **The case for it is strong** ‚Äî every bug found this release cycle
  (the self-tail loop, the legacy-schema migration, the `ATTACH` limit)
  was found by actually running grepod against a real cluster, not by
  unit tests. That's real evidence an e2e smoke test would catch things
  unit tests structurally can't.
- **But this session can't respect this repo's own SHA-pinning
  convention for it.** Wiring in `kind`/`helm` setup means either adding
  a new third-party Action (which `CLAUDE.md`'s conventions require
  SHA-pinning, the way `.github/workflows/*.yml` already does ‚Äî not
  guessable/fabricatable without network access to verify current SHAs
  from this environment) or hand-rolling binary installs via `curl`
  (verifiable version numbers, but no way to test the resulting workflow
  actually works without pushing it and watching CI run, which wasn't
  available here either).
- Shipping a CI workflow that was never actually run, in a repo whose own
  `RELEASE.md` process leans hard on "verify locally before claiming
  done," would violate that same discipline. Recommendation for whoever
  picks this up: mirror `scripts/local-test.sh` almost exactly, add
  `pinact`-verified SHA pins for `helm/kind-action` (or equivalent), mark
  the job `continue-on-error: true` so it's informational rather than
  gating, and ‚Äî critically ‚Äî actually watch it run green in CI at least
  once before merging, not just review the YAML.
