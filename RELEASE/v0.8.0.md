# v0.8.0 — Read/write timeouts

**Status:** not started.

## Goal

`cmd/server/main.go`'s `http.Server` only sets `ReadHeaderTimeout: 10 *
time.Second`. Go's own `http.Server` docs call this out explicitly: without
`ReadTimeout`/`WriteTimeout`/`IdleTimeout`, a slow client can still hang a
connection indefinitely after headers are read (a slow request body, or a
slow reader on the response side) — `ReadHeaderTimeout` alone doesn't cover
that. Separately, `storage.Store.Search` takes no `context.Context` at all
today: it calls `sql.DB.Query`/`sql.DB.Exec`, not the `...Context` variants,
so a request's timeout/cancellation never actually reaches the database —
an expensive cross-shard query has nothing bounding it but whatever the new
`WriteTimeout` eventually enforces from the outside.

## Planned scope

- Set `ReadTimeout`, `WriteTimeout`, and `IdleTimeout` on the `http.Server`
  in `cmd/server/main.go`, sized for this workload (a small JSON API plus
  one static page — `WriteTimeout` needs enough headroom for a genuinely
  large cross-shard search, not just a typical fast response).
- Thread `context.Context` through `storage.Store`: `Search(ctx, ...)` and
  `InsertBatch(ctx, ...)` using `QueryContext`/`ExecContext` instead of
  the context-less variants, so a request's deadline (or an early client
  disconnect) actually cancels the underlying SQLite query instead of
  leaving it to run to completion regardless.
- `internal/api.Handler.handleSearch` passes `r.Context()` into `Search`
  instead of the timeout only being enforced by the outer `WriteTimeout`.
- Decide whether `BatchQueue`'s writer path also gets a bounded
  per-flush context (protects against one pathological shard write
  stalling ingestion) or whether that's out of scope here — ingestion
  isn't request-driven the way search is, so a hang there has different
  failure characteristics than a slow HTTP response.
- Extend the v0.2.0 test suite: a `Search` call with an already-cancelled
  context returns promptly with a context error rather than running the
  query to completion.
- Update [DESIGN/04](../DESIGN/04_design_api/01_search.md) to document the new
  timeout values and that `/api/search` now honors client disconnects.

## Goal (write-side SQLite concurrency)

SQLite tolerates one writer at a time well, but not a high rate of small,
frequent write transactions from concurrent goroutines — `getOrOpenDB`
already pins `SetMaxOpenConns(1)` per shard for exactly this reason.
`BatchQueue`'s current defaults (`BATCH_SIZE=200` or `BATCH_INTERVAL=500ms`,
whichever first) still flush fairly often under a busy namespace. Widen
the consolidation window so more lines land in a single transaction.

## Planned scope (write-side SQLite concurrency)

- Change the default flush cadence to consolidate in memory and insert
  roughly every 15s, rather than the current 500ms default — fewer,
  larger transactions per shard instead of many small ones. `BATCH_SIZE`
  stays as the other trigger (still flush early if the buffer fills), but
  the time-based default moves from `500ms` to `15s`.
- Re-check `BatchQueue.Enqueue`'s channel capacity (`size*4`) against the
  new interval — a longer consolidation window means more lines can be
  in-flight between flushes, so the drop-when-full path
  (`grepod_lines_dropped_total`, from [v0.7.0](v0.7.0.md)'s metrics work)
  needs headroom sized for 15s of ingestion at a realistic line rate, not
  500ms of it.
- Weigh the latency trade-off explicitly in
  [DESIGN/03](../DESIGN/03_design_storage.md): a line enqueued right after
  a flush is now searchable up to ~15s later instead of ~500ms later.
  Document this as the expected search-latency floor, not a bug.
- Extend the v0.2.0 test suite's `BatchQueue` tests for the new default,
  and confirm `TestBatchQueue_FlushesOnSize` still exercises the
  size-triggered path independently of the larger interval.
