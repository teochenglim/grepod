# v0.5.0 — Live tail UI + search UX

**Status:** shipped, tagged `v0.5.0`.

## Goal

Wire [v0.4.0](v0.4.0.md)'s `/api/tail` into the UI, and fix the two search
UX limits that become more noticeable once people are actually using this
day to day: no way to browse past the 500-result cap, and no way to
narrow by pod/container without knowing the exact name to type.

## Delivered scope

- **Tail view** in `web/templates/index.html` / `web/static/app.js`:
  toggle between "Search" (existing, historical) and "Tail" (new, live)
  modes, implemented as two sibling `<section>`s with `hidden` toggled by
  JS rather than separate pages. Tail mode connects an `EventSource` to
  `/api/tail`, auto-scrolls, and turns that off the moment the user
  scrolls up — new lines keep arriving and appending underneath (pausing
  never drops lines), with a "Resume" button to jump back to the bottom
  and re-enable it. Client-side buffer capped at 2000 lines (oldest
  evicted first) so a long session doesn't grow the DOM unboundedly.
- **Pod/container filters** in Tail mode: `<select>` dropdowns populated
  from `GET /api/known` (new endpoint, backed by new
  `Store.KnownPods(since time.Time) (KnownFilters, error)`), instead of
  requiring an exact free-text match against `/api/tail`'s existing
  `pod`/`container` params.
- **Pagination** past the hardcoded 500-result cap in
  `storage.Store.Search`: `Search` now takes a `SearchOptions` struct and
  returns a `SearchPage{Results, NextCursor}`. The cursor is a keyset over
  the query's actual sort order (`bm25()` rank, then shard, then the
  per-shard FTS5 `rowid` as a tiebreaker), base64-encoded and opaque to
  the caller — chosen over `OFFSET`, which would have to walk and discard
  every prior row on every attached shard on every page. A "Load more"
  button in the UI appends each new page to what's already rendered.
- **Level-based filtering**, built on [v0.3.0](v0.3.0.md)'s `level`
  column (query-param work plus one `Search`/`KnownPods`-shaped addition
  to `storage` — no further schema changes):
  - **Level tabs**: `ALL / INFO / WARN / DEBUG / FATAL` above the results
    list; clicking one re-runs the search scoped to that level (via the
    new `level` query param) without retyping the query.
  - **"This level and anything more severe"**: `storage.levelOrder`
    (`FATAL > ERROR > WARN > INFO > DEBUG > TRACE`) plus
    `levelsAtOrAbove(minLevel)` turn a tab selection into the set of
    levels `Search` filters on with `level IN (...)` per shard — so
    `level=WARN` returns `WARN`, `ERROR`, and `FATAL`. An empty or
    unrecognized `level` column is its own bucket: visible under "ALL",
    but never matches a specific tab, since there's no way to rank it.
  - **Group occurrences, reverse display**: a client-side aggregation
    (`app.js`'s `groupResults`) over the accumulated result set, not a new
    SQL aggregate query. Groups by `pod + container +
    normalizeForGrouping(snippet)`, where the latter strips `<mark>` tags
    and replaces UUIDs/ISO timestamps/bare numbers with a placeholder so
    e.g. repeated errors differing only by request ID collapse into one
    `×N` entry, sorted most-recent-first.
- Updated [DESIGN/04](../DESIGN/04_design_api.md) for `/api/known`, the
  `level`/`cursor` params, and the `next_cursor` response field.

## Verified

`gofmt -l .` (clean), `go vet ./...`, `go test ./... -race -cover`
(47/47; `internal/api` 86.1%, `internal/storage` 83.1%). Backend exercised
end-to-end over real HTTP (not just `httptest.NewRecorder`): a throwaway
`net/http` server seeded with sample pods/levels/repeated messages,
`curl`-verified `/api/known`, `/api/search` with `level=WARN` and the
`ALL` default, `/api/tail` SSE streaming, and that `/` serves the new
markup (`modeSearchBtn`, `levelTabs`, `tailPod`/`tailContainer`,
`loadMoreBtn`, `tailResumeBtn`) with `app.js` passing a Node syntax check.
No cluster available in this environment to drive the UI through an
actual browser against a live `client-go` tailer — a human should click
through Search↔Tail switching, the level tabs, "Load more", and the
group-occurrences toggle against a real deployment before wide rollout.
