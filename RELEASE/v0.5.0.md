# v0.5.0 — Live tail UI + search UX

**Status:** not started.

## Goal

Wire [v0.4.0](v0.4.0.md)'s `/api/tail` into the UI, and fix the two search
UX limits that become more noticeable once people are actually using this
day to day: no way to browse past the 500-result cap, and no way to
narrow by pod/container without knowing the exact name to type.

## Planned scope

- **Tail view** in `web/templates/index.html` / `web/static/app.js`:
  toggle between "Search" (existing, historical) and "Tail" (new, live)
  modes. Tail mode auto-scrolls with a pause-on-scroll-up affordance (a
  common pattern for log viewers — pausing shouldn't drop lines, just
  stop auto-scrolling to them) and a resume button.
- **Pod/container filters** — a dropdown populated from the distinct
  `pod`/`container` values seen recently (a small addition to
  `storage.Store`, e.g. `Store.KnownPods(since time.Time)`), instead of
  requiring an exact free-text match.
- **Pagination** past the current hardcoded 500-result cap in
  `storage.Store.Search` — cursor-based (on `rowid` or timestamp) rather
  than `OFFSET`, since `OFFSET` gets slow across `UNION ALL`-ed shards at
  scale.
- **Level-based filtering**, built on [v0.3.0](v0.3.0.md)'s `level`
  column (all client-side/query-param work — no further schema changes):
  - **Level tabs**: `ALL / INFO / WARN / DEBUG / FATAL` above the results
    list, so e.g. clicking `WARN` re-runs the search scoped to that level
    without retyping the query.
  - **"Effectively filter the search term and above"**: selecting a tab
    like `WARN` doesn't mean *only* `WARN` — it means `WARN` and anything
    more severe (`WARN`, `ERROR`, `FATAL`), the same convention as most
    log-level filters. Needs a severity ordering, not just an equality
    filter, since `level` is stored as free text
    (`FATAL > ERROR > WARN > INFO > DEBUG > TRACE`, empty/unrecognized
    treated as its own bucket rather than silently dropped).
  - **Group occurrences, reverse display**: when many rows share the same
    message (common for repeated errors/warnings), collapse them into one
    entry with a count, ordered most-recent-first — needs a grouping key
    (message text, or message-minus-obvious-variable-tokens like
    timestamps/IDs — decide at implementation time how fuzzy the grouping
    should be) and is naturally a client-side aggregation over an already
    date-ranged/leveled result set, not a new SQL aggregate query.
- Update [DESIGN/04](../DESIGN/04_design_api.md) for the new query
  params/response shape.
