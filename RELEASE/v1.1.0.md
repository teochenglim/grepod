# v1.1.0 — Pod filter dropdown for Search

**Status:** ready to tag — implemented and verified locally, uncommitted.
Sequenced after `v1.0.0`: don't `make release VERSION=1.1.0` until
`v1.0.0` is actually tagged first (a real `git tag`, not just its doc
saying "shipped") — releases land in order.

## Goal

The Tail view has had a pod/container filter dropdown since
[v0.5.0](v0.5.0.md), but the Search view had no equivalent — narrowing a
search to one pod meant typing its name into the FTS5 query itself (and
FTS5 has no notion of "and also only this pod," so that didn't actually
work; there was no way to do it at all). Requested directly: a pod
dropdown on the Search toolbar, defaulting to "All pods," same UX as
Tail's.

## Delivered scope

- **`storage.SearchOptions` gained `Pod string`** (`""` = no filtering,
  the default). `Store.Search` applies it as an exact match in both
  browse and keyword mode, combined with `MinLevel` via `AND` — see
  [DESIGN/03](../DESIGN/03_design_storage.md#search-cross-shard-attach)
  for how the two filters share one conditions-building path instead of
  each hand-rolling `WHERE`-vs-`AND`.
- **`/api/search` gained a `pod` query param** (exact match, `""` = no
  filtering) and echoes it back in the response (`searchResponse.Pod`) —
  see [DESIGN/04/01_search](../DESIGN/04_design_api/01_search.md). This is
  an additive change under [COMPATIBILITY.md](../COMPATIBILITY.md)'s own
  policy (new query params/response fields are fine in a minor release),
  so it doesn't conflict with v1.0.0's compatibility commitment shipped
  immediately before this.
- **UI**: a `<select id="searchPod">` added to the Search toolbar, right
  before the date inputs — same position/pattern as Tail's pod dropdown.
  Populated from the existing `/api/known` call (`loadKnownFilters` now
  populates both dropdowns from one fetch instead of just Tail's).
  Selecting a pod re-runs the search immediately (`change` listener), same
  as the level tabs.
- New tests: `TestSearch_PodFiltersToExactMatch` (keyword mode, browse
  mode, and no-filter subtests), `TestSearch_PodAndLevelFiltersCombine`
  (`internal/storage`); `TestHandleSearch_PodFiltersToExactMatch`
  (`internal/api`).

## Verified

`gofmt -l .` (clean), `go vet ./...`, `go build ./...`, and
`go test ./... -race -count=1` (all packages green, including the 3 new
tests). UI verified end-to-end, not just unit-tested: built a throwaway
`internal/api.Handler` against a `storage.Store` seeded with lines across
two pods (`web-1`, `web-2`), served it locally, and drove it with
Playwright — confirmed the dropdown renders before the date inputs,
populates as `All pods`/`web-1`/`web-2`, and selecting `web-1` narrows 4
results down to exactly web-1's 2 lines (screenshotted both states). The
throwaway program and its data dir were deleted after, never committed.
Also confirmed manually by the user in their own browser.
