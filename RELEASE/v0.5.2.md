# v0.5.2 — Fix permanent ingestion failure; browse logs without a keyword; fix unescaped log content in the UI

**Status:** ready to tag — implemented and verified locally, uncommitted.

## Goal

Everything in this release came from actually running [v0.5.1](v0.5.1.md)
against a real cluster (`make k8s-apply`) and troubleshooting live, rather
than from planned scope:

1. **Permanent ingestion failure** on a shard whose schema predates the
   `level` column.
2. **`make help` silently omitting `k8s-apply`/`k8s-delete`/`k8s-logs`**
   from its target listing (noticed while using it to drive the rest of
   this troubleshooting).
3. **Search requiring a keyword just to see anything**, discovered while
   confirming live tail worked but search looked "empty."
4. **Unescaped log content reaching the browser as real HTML**, found
   while fixing (3) — not introduced by it, but made worse by it (browse
   mode would have shipped a second code path with the same gap).

## Bugs found and fixed

### 1. Permanent ingestion failure on a legacy-schema shard

```json
{"level":"ERROR","msg":"failed to flush log lines","pod_namespace":"default","pod_name":"grepod-78957b869d-szcwr","count":1,"err":"insert into shard 2026-07-12: SQL logic error: table fts has no column named level (1)"}
```

`internal/storage/sqlite.go`'s `schema` constant is `CREATE VIRTUAL TABLE
IF NOT EXISTS fts USING fts5(... level UNINDEXED ...)` — but `IF NOT
EXISTS` is a no-op against a shard file that already has an `fts` table.
A shard created before v0.3.0 added the `level` column (documented as a
breaking, pre-1.0 change in [DESIGN/03](../DESIGN/03_design_storage.md) —
"delete and re-ingest") keeps its original 5-column schema forever, and
every subsequent `INSERT ... (pod, namespace, container, timestamp,
level, line)` into it fails outright, not just that shard's historical
searchability. Concretely: on a long-lived local dev PVC reused across
many versions, "today's" shard can be one of these legacy files, meaning
*all* ingestion for the whole namespace silently fails until the next UTC
day rolls over a fresh shard — a correctness bug well beyond what
"delete and re-ingest old data" was ever meant to describe.

The obvious fix — `ALTER TABLE fts ADD COLUMN level` — doesn't exist:
FTS5 virtual tables reject `ALTER TABLE` outright (`"virtual tables may
not be altered"`, confirmed directly against `modernc.org/sqlite` before
committing to an approach). `DROP TABLE` + recreate is the only option.

- **`getOrOpenDB` now detects and migrates a legacy-schema shard** on
  first open for a write: `SELECT count(*) FROM pragma_table_info('fts')
  WHERE name = 'level'` — if the column is missing, `DROP TABLE fts`
  followed by recreating it under the current schema, logging a warning
  naming the shard. This loses that shard's pre-migration rows (same
  delete-and-re-ingest stance the project already takes on pre-1.0 schema
  changes), but a shard-local, automatic, once-per-shard rebuild is a
  strict improvement over every future insert into it failing forever.
- **Known remaining gap, explicitly not fixed here**: `Store.Search`
  attaches shard files directly via a per-query `ATTACH DATABASE`,
  bypassing `getOrOpenDB` (and its migration) entirely. A legacy shard
  that's searched but never written to again still fails that `SELECT`
  with `"no such column: level"`. Fixing this would mean running the same
  migration against a connection Search doesn't otherwise need a
  write-capable handle for, and risks lock contention with a concurrently
  open write connection for the same date (`Store.dbs[date]`,
  `SetMaxOpenConns(1)`) — judged not worth the added complexity for a
  read-only, one-shot failure mode, versus the write-path bug above, which
  silently and indefinitely broke ingestion. Manual delete-and-re-ingest
  remains the answer for that case. See
  [DESIGN/03](../DESIGN/03_design_storage.md#store-daily-sharded-sqlite--fts5).

### 2. `make help` silently omitting targets with digits in their name

`^[a-zA-Z_-]+:.*?## .*$` (the regex `help` uses to find `##`-documented
targets) has no `0-9` in its character class, so `k8s-apply`, `k8s-delete`,
and `k8s-logs` never matched — they worked fine when run directly, they
just never appeared in the menu. Fixed: `^[a-zA-Z0-9_-]+:.*?## .*$`.

### 3. Search required a keyword just to see anything

The Search view showed "Type a keyword first." until you typed something,
even though live tail already worked with no filters — discovered while
confirming live tail worked but search looked "empty." (Ingestion was
actually current the whole time — confirmed via a direct SQLite query
against the mounted PVC, `max(timestamp)` within 6s of wall-clock; the
"nothing shows up" impression was purely this UI/API gate, not a data
problem.)

- **`q` is now optional on `/api/search`.** Empty or whitespace-only `q`
  is browse mode, not a `400` and not "search for nothing": every line in
  `[start, end]` (still narrowable by `level`), most-recent-first, no FTS5
  `MATCH` involved (`bm25()`/`snippet()` only mean something with an
  active `MATCH`, so browse-mode results carry the raw line and `rank:
  0`). A keyword still narrows the same view — it's an optional filter on
  top of browsing, not a gate on seeing anything at all. See
  [DESIGN/04](../DESIGN/04_design_api.md#browse-mode-v053).
- **`Store.Search` cursor pagination extended to browse mode**: a second
  keyset ordering (shard, then per-shard FTS5 `rowid`, both descending —
  recency, since there's no relevance rank to order by) alongside the
  existing bm25-rank ordering for keyword search, sharing the same cursor
  encoding (mode-tagged) and outer pagination-wrapping SQL. See
  [DESIGN/03](../DESIGN/03_design_storage.md).
- **`web/static/app.js`**: removed the "type a keyword first" guard; runs
  an initial browse-mode search on page load so there's something to look
  at immediately; level tabs and the query box both just re-run the same
  (possibly keyword-less) search.

### 4. Unescaped log content reaching the browser as real HTML

Found while implementing (3) — building the browse-mode code path meant
looking closely at how `snippet` reaches the browser. SQLite's
`snippet()` wrapped only the matched term in `<mark>`/`</mark>` — the
rest of the snippet (all of it, in browse mode, since there's no match)
was the **raw log line**, and the UI injects `snippet` via `innerHTML` to
render that highlighting. A log line containing real markup (a raw
request path or header logged verbatim, containing e.g.
`<img src=x onerror=...>`) would execute as HTML in the browser — a
stored-XSS vector reachable by anything whose logs grepod ingests, i.e.
every pod in the namespace. Predates this release (existing keyword
search already had it); fixed here rather than shipping browse mode as a
second code path with the same gap.

- **Fix**: `snippet()` now emits sentinel control bytes (`\x01`/`\x02`)
  instead of literal `<mark>`/`</mark>`; the entire result is
  HTML-escaped server-side (`html.EscapeString`, `storage.escapeSnippet`),
  then the sentinels are substituted for the real tags — so the *only*
  real markup that can ever reach the browser is highlighting grepod
  itself added, never anything from log content.

## Delivered scope (non-bug)

- `Store.KnownPods`/`Search` unaffected in shape; `SearchResult.Rank` is
  simply `0` in browse-mode results (documented, not a bug — bm25 has no
  meaning without an active `MATCH`).
