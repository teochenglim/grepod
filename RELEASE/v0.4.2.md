# v0.4.2 — Release tooling fix: push HEAD before tagging

**Status:** shipped, tagged `v0.4.2`.

## Goal

[v0.4.1](v0.4.1.md) removed `git push origin HEAD` from `make release`
entirely, on the theory that pushing your own commits was your
responsibility, done separately. In practice, cutting `v0.4.1` itself hit
exactly the risk that release's own doc had flagged and not yet been
observed for real: the commit `v0.4.1` pointed to had been committed
locally but never pushed to `origin/main` before `make release` ran.
`git push origin v0.4.1` still succeeded and still triggered CI (pushing
a tag pushes whatever commit object it points to, regardless of whether
that commit is reachable from any branch on the remote) — but GitHub's UI
couldn't display the commit's message in the Actions run list, because
the commit wasn't part of `main`'s history there. It rendered as a bare
"by teochenglim" instead of the real commit subject, unlike `v0.3.0`
(tagged after its commit was already on `origin/main`), which showed the
full message correctly.

## Delivered scope

- **`make release` now runs `git push origin HEAD` before creating the
  tag**, not after (and not never, per v0.4.1). Order matters: pushing
  first guarantees the commit is already part of `origin/main`'s history
  by the time the tag — and the CI run it triggers — exists, so GitHub
  can always resolve and display the real commit message.
- This does **not** reopen what v0.4.1 fixed: `release` still never
  creates a commit itself. It only pushes whatever you already committed
  — same "you commit, `make release` handles push+tag" division of
  responsibility, just with the push now happening automatically instead
  of being a separate manual step you could forget (as happened here).
- Same fix applied to `~/code/servicedesk`'s `Makefile`.

## Bugs found and fixed

- The exact failure mode v0.4.1's own release note predicted
  ("isn't visible as part of `main`'s history on GitHub until you push
  the branch too") happened for real on `v0.4.1`'s own release, one
  release later — confirmed by `git status` showing `Your branch is
  ahead of 'origin/main' by 1 commit` and `git branch -r --contains
  v0.4.1` returning empty (the tagged commit reachable from no remote
  branch). Root cause matched the prediction exactly; fix is to stop
  relying on a separate manual step for something `release` can just
  always do correctly itself, in the right order.

## Verified

`make -n release VERSION=x.y.z` dry-run reviewed for both grepod and
servicedesk, confirming `git push origin HEAD` now precedes `git tag`/
`git push origin <tag>` in the printed command sequence.
