# v0.7.0 — Restart-safe tailing

**Status:** not started.

## Goal

Discovered while writing [v0.2.0](v0.2.0.md)'s test suite and confirmed by
reading `tailContainer`/`streamLogs`: grepod has no persisted marker for
"how far into a container's log I've already ingested." Every time a
tailer (re)starts — a fresh pod, a restart-count bump, or grepod itself
restarting and re-listing every pod on startup — `streamLogs` calls
`GetLogs` with `Follow: true` and no `SinceTime`/`SinceSeconds`/
`TailLines`. Per the Kubernetes API, that combination returns the
container's **entire currently-buffered log content** first, then follows
live (the same behavior as `kubectl logs -f` with no `--tail`). So on
every grepod restart, every live container's full current log buffer gets
re-ingested into SQLite — duplicate rows for anything still sitting in
that buffer. Not data loss, but real duplication, wasted ingestion work,
and confusing duplicate hits in search results.

## Planned scope

- Track a per-container "last ingested timestamp" — in memory is enough
  for the common case (grepod restarting), since a fresh `Manager` has no
  prior state anyway; the goal is to stop *within-buffer* re-ingestion on
  reconnect/restart, not to survive `/data` itself being wiped (that's
  what retention already governs).
- Actually, in-memory alone doesn't help across a full grepod process
  restart — the state needs to survive that. Two options to weigh at
  implementation time:
  1. Persist a small "last seen timestamp per pod/container" table in
     `storage.Store` (a new non-FTS table, separate from the `fts` shards)
     that `tailer.Manager` reads on startup and writes periodically.
  2. Derive it from the data already in SQLite — query the max
     `timestamp` already indexed for a given pod/container before
     starting its stream, and pass that as `SinceTime`.
     Simpler (no new schema), but adds a query per container on every
     (re)start/reconnect — check it doesn't materially slow down startup
     on a namespace with many pods.
- Pass the resolved marker as `PodLogOptions.SinceTime` on `streamLogs`
  (not `fetchPreviousLogs` — the crash-recovery path is unrelated and
  already correctly bounded by `TailLines: 100`).
- Decide and document the behavior for a fresh container with no prior
  marker (the common case: fall through to today's behavior — ingest
  everything currently buffered, since there's nothing to dedupe against
  yet).
- Extend the v0.2.0 test suite: a tailer restarting for a container with
  a known last-ingested timestamp does not re-enqueue lines at or before
  that timestamp.
- Update [DESIGN/02](../DESIGN/02_design_tailer.md) — this changes a
  documented behavior, not just adds one.
