**Status:** shipped, tagged `v0.7.0`.

## Goal (restart-safe tailing)

Discovered while writing [v0.2.0](v0.2.0.md)'s test suite and confirmed by
reading `tailContainer`/`streamLogs`: grepod has no persisted marker for
"how far into a container's log I've already ingested." Every time a
tailer (re)starts â€” a fresh pod, a restart-count bump, or grepod itself
restarting and re-listing every pod on startup â€” `streamLogs` calls
`GetLogs` with `Follow: true` and no `SinceTime`/`SinceSeconds`/
`TailLines`. Per the Kubernetes API, that combination returns the
container's **entire currently-buffered log content** first, then follows
live (the same behavior as `kubectl logs -f` with no `--tail`). So on
every grepod restart, every live container's full current log buffer gets
re-ingested into SQLite â€” duplicate rows for anything still sitting in
that buffer. Not data loss, but real duplication, wasted ingestion work,
and confusing duplicate hits in search results.

## Delivered scope (restart-safe tailing)

Went with option 2 from the two weighed above (derive from what's already
in SQLite), but not literally "a query per container on every
(re)start/reconnect" â€” that concern is what shaped the actual design:

- `tailer.Manager` tracks an in-memory `marker` per `containerKey` (the
  timestamp of the most recently ingested *live* line for that container),
  passed as `PodLogOptions.SinceTime` on every `streamLogs` call for that
  container â€” including the first. `fetchPreviousLogs` is untouched, as
  planned: it's a different, already-crashed container instance, correctly
  bounded by `TailLines: 100` already, and must never influence the
  current instance's `SinceTime`.
- New `storage.Store.LastSeen(pod, container) (time.Time, bool, error)`
  seeds the marker, but `tailer.Manager.resolveMarker` only calls it
  **once per container per process lifetime**, caching the result â€”
  addressing the "adds a query per container on every reconnect" concern
  head-on rather than accepting it. Every reconnect within a process's
  lifetime (dropped stream, informer resync) reuses the cached in-memory
  value; only a fresh process discovering an already-running container
  ever hits SQLite. `LastSeen` itself scans shards most-recent-day-first
  and stops at the first match (shards are disjoint by date, so the first
  hit is already the max), bounded to 30 days back.
- No new schema â€” went with plain `SELECT MAX(timestamp) FROM fts WHERE
  pod = ? AND container = ?` against each shard file directly (its own
  short-lived connection, separate from `Store.dbs`' write handles),
  scoped to option 2 as planned. Turned out unneeded for the
  restart-count-bump case: a container restart gets a *fresh* log buffer
  from Kubernetes (nothing to dedupe there), so only the fresh-process and
  dropped-stream cases actually needed the marker at all â€” the original
  goal's "restart-count bump" framing overstated the problem slightly, see
  [DESIGN/02](../DESIGN/02_design_tailer.md#restart-safe-reconnects-v070)
  for the corrected description.
- A container with no marker (never indexed, or `LastSeen` found nothing
  within the 30-day lookback) falls through to the pre-v0.7.0 behavior â€”
  no `SinceTime`, ingest whatever's currently buffered â€” exactly as
  planned.
- Markers are cleared on an actual pod delete (`stopPod`) but deliberately
  preserved across a restart-count-triggered stop/start
  (`reconcilePod`'s restart path), so the fresh goroutine started right
  after a crash still benefits from what the crashed instance ingested.
- **Known imprecision, documented rather than hidden:** the marker is
  grepod's own ingestion-time stamp (`time.Now()` at scan time), not the
  container runtime's actual per-line timestamp that `SinceTime` filters
  on server-side â€” consistent with `ingest`'s pre-existing stance that
  ingestion time, not emission time, is what grepod indexes throughout.
  Close enough in practice that this only matters at the exact boundary
  line (which the K8s API includes inclusively), not a source of real
  duplication or loss.
- `DESIGN/02` updated with a new "Restart-safe reconnects (v0.7.0)"
  section; `DESIGN/03` gained a `LastSeen` section.
- New tests: `TestLastSeen_ReturnsMostRecentTimestampForContainer`,
  `TestLastSeen_NoRowsReturnsNotFound`,
  `TestLastSeen_FindsMarkerInPriorDayShard` (`internal/storage`);
  `TestIngest_TracksMarkerOnlyWhenRequested`,
  `TestResolveMarker_QueriesStoreOnceThenCaches`,
  `TestResolveMarker_NoStoreMatchFallsThroughToZeroMarker`,
  `TestResolveMarker_NilStoreDoesNotPanic`,
  `TestMarkers_ClearedOnPodDeleteButNotOnContainerRestart`
  (`internal/tailer`).

## Goal (RED metrics)

Moved here from [v0.3.0](v0.3.0.md) deliberately: RED (Rate, Errors,
Duration) metrics are most useful once every critical point they'd cover
actually exists â€” insert (`v0.1.0`), tail/streaming (`v0.4.0`/`v0.5.0`),
and query (`v0.1.0`, refined in `v0.5.0`'s pagination). Landing `/metrics`
here, after that work, means the metric set is designed against the real
shape of the system instead of guessed at ahead of it.

## Delivered scope (RED metrics)

- **`/metrics`** â€” `promhttp.Handler()`, registered in `api.New` like any
  other route (same port as everything else). One new direct dependency,
  `github.com/prometheus/client_golang` (`v1.20.5`), as planned; its own
  transitive deps (`prometheus/common`, `prometheus/procfs`,
  `klauspost/compress`) came along with it. New leaf package
  `internal/metrics` defines every collector â€” `tailer`, `storage`, and
  `api` each depend on it (never the reverse), keeping
  [ARCHITECTURE.md](../ARCHITECTURE.md)'s downward/inward dependency rule
  intact with one package added. Every collector registers against the
  default Prometheus registry, so `/metrics` also surfaces
  `client_golang`'s free process/Go-runtime metrics.
  - **Insert** (`storage.BatchQueue.flush`): `grepod_insert_requests_total`,
    `grepod_insert_errors_total`, `grepod_insert_duration_seconds`
    (histogram), `grepod_lines_dropped_total` (from `recordDrop`,
    independent of its rate-limited log warning) â€” all as planned.
  - **Tail** (`tailer.Manager.tailContainer`): `grepod_tail_streams_total`,
    `grepod_tail_stream_errors_total` â€” as planned. `grepod_tail_subscribers`
    shipped as a `promauto.NewGaugeFunc` in `cmd/server/main.go` reading
    `Broadcaster.SubscriberCount()` on every scrape, rather than
    `Broadcaster` incrementing/decrementing inline â€” keeps `Broadcaster`
    itself free of any Prometheus dependency.
  - **Query** (`api.Handler.handleSearch`): `grepod_search_requests_total`,
    `grepod_search_errors_total`, `grepod_search_duration_seconds`
    (histogram) â€” as planned.
  - Every metrics-holding struct (`BatchQueue`, `Manager`, `Handler`)
    accepts a `*metrics.Metrics` and nil-checks it before use, since
    several existing tests build these types via struct literal, bypassing
    the constructor, and must not be forced to wire up a registry.
- **Deviation from the plan on k8s/helm**: no separate `metrics`
  container port â€” grepod is one HTTP server, so a second port would just
  duplicate the existing one under a different name. Instead,
  `20-deployment.yaml`/`templates/deployment.yaml` always set
  `prometheus.io/scrape`/`port`/`path` pod annotations (works with the
  classic annotation-based scrape config, zero cluster prerequisites), and
  a new `helm/templates/servicemonitor.yaml`, gated behind
  `serviceMonitor.enabled` (**default `false`**, since enabling it without
  the Prometheus Operator's CRDs installed fails `helm install` validation),
  covers the Operator case. `helm lint`/`helm template` (both the default
  and `--set serviceMonitor.enabled=true`) verified clean.
- Full metric list documented in a new "Metrics (v0.7.0)" section in
  [DESIGN/04](../DESIGN/04_design_api.md#metrics-v070); `helm/README.md`
  and `k8s/README.md` each gained a "Metrics" section.
