# v0.7.1 — Release tooling: fold the version bump into the release commit

**Status:** shipped, tagged `v0.7.1`.

## Goal

Every release since [v0.4.1](v0.4.1.md) left `VERSION`, `helm/Chart.yaml`,
and `k8s/20-deployment.yaml`'s image tag rewritten-but-uncommitted after
`make release` — deliberate at the time (see v0.4.1's four-iteration
history: no auto-commit, so the tag always lands on the commit you
actually wrote, not a generic bump message), but "commit it yourself
whenever convenient" turned into "forget to, every single time" in
practice (surfaced again cutting [v0.7.0](v0.7.0.md)). Goal here: fold
those three files into the *same* commit the tag lands on, without
reintroducing the original problem (a generic bump message ending up on
the tag).

## Delivered scope

- `make release`'s target gained two lines, right after the inline `bump`
  call and before the push:
  ```
  $(MAKE) bump VERSION=$(VERSION)
  git add VERSION helm/Chart.yaml k8s/20-deployment.yaml
  git commit --amend --no-edit
  git push origin HEAD
  git tag v$(VERSION)
  git push origin v$(VERSION)
  ```
  `--no-edit` keeps whatever message the preceding commit already had —
  same property v0.4.1 established (the tag's commit message is always
  the real one you wrote, never a generic bump line), just with the bump
  files folded in instead of left dangling.

## Known limitation: depends on the commit still being unpushed

`git commit --amend` rewrites the commit's SHA. That's only safe when the
commit being amended hasn't been pushed anywhere yet — amending an
already-public commit diverges local history from the remote, and the
`git push origin HEAD` two lines later then fails non-fast-forward (or
would need a force-push to "fix," which risks corrupting whatever anyone
else already fetched, including an already-published tag pointing at the
pre-amend SHA).

**`RELEASE.md`'s documented pre-step still tells you to `git push origin
main` before running `make release`** — i.e., exactly the sequence that
makes this amend unsafe. That contradiction was flagged during this
release and deliberately left unresolved at the user's request (not a
gap I missed): fixing it would mean changing `RELEASE.md`'s instructions
to stop pushing before `make release` runs, which wasn't wanted yet.
**Until that's decided, don't `git push origin main` before `make
release`** — commit locally, then run `make release`, and let it do the
one-and-only push itself.

This surfaced concretely while cutting this exact release: `v0.7.0`'s
commit had already been pushed and tagged before this change was tested,
so amending it locally diverged `main` from `origin/main` (`ahead 1,
behind 1`) rather than cleanly folding in. No force-push was performed —
see "Verified" below for how this was actually resolved for v0.7.1's own
tag.

## Verified

`make -n release VERSION=x.y.z`-equivalent read-through of the new
target; no automated test (this is Makefile/git-process tooling, not
application code — same category as v0.4.1/v0.4.2, verified by dry-run
and reasoning about the sequence, not `go test`). `go build`/`go vet`/
`go test ./... -race` unaffected (no application code touched) and not
re-run for this release.
